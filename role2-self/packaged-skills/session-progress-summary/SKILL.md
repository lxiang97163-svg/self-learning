# 会话进度总结

## 目标

把当前对话窗口的上下文，以及本轮对话中相关的已生成或已修改文件，整理成一份可交接、可继续推进的 Markdown 文档。

输出必须回答这 4 类问题：

1. 这次对话整体在做什么，核心问题是什么。
2. 当前进度如何，哪些已经解决，哪些还没解决。
3. 未解决部分下一步怎么做；如果涉及代码，给出可选技术方案并明确推荐方案。
4. 把结果写到工作区根目录，文件名格式为“问题 + 日期 + 版本号”。

## 默认范围

默认总结范围是：

- 当前对话窗口
- 本轮对话中明确提到、创建、修改、审阅过的相关文件

若用户明确要求扩大或缩小范围，再按用户要求调整。

## 执行流程

### Step 1：确认总结对象

先从当前对话中提炼 3 个要点：

- 当前在推进的主问题
- 关键决策或已确认约束
- 涉及的文件、模块、命令或方案

如果“问题名”不够明确，优先根据对话主线压缩成一个简短标题；只有在完全无法判断时才询问用户。

问题名要求：

- 尽量短，适合放进文件名
- 优先使用用户原话中的问题表述
- 避免使用“这个问题”“当前事项”这类空泛标题

### Step 2：判断进度状态

把内容拆成以下 4 个部分来判断：

1. 已完成：
   - 已实现
   - 已验证
   - 已确认结论
2. 进行中：
   - 已开始但未收尾
   - 已有方向但还没执行完
3. 未解决：
   - 尚未实现
   - 尚未验证
   - 仍存在阻塞、风险或待确认项
4. 下一步：
   - 最合理的继续推进顺序

不要把“讨论过”误写成“已解决”。

### Step 3：分析未解决部分

对每个未解决项，至少补充以下内容：

- 当前阻塞点是什么
- 下一步思路是什么
- 若是代码相关问题：列出 2 到 3 种可行技术方案
- 明确推荐其中一种，并说明推荐原因

推荐方案时优先考虑：

- 与现有代码风格和架构一致
- 改动范围可控
- 回归风险较低
- 验证路径清晰

若某个问题其实不需要多方案比较，直接给出单一建议即可，不必为了形式硬凑多个方案。

### Step 4：生成根目录文档

将总结写入当前工作区根目录，文件名格式：

`<问题名>-<YYYYMMDD>-v<序号>.md`

示例：

- `支付回调修复-20260408-v1.md`
- `Xboard节点接入-20260408-v2.md`
- `会话总结-20260408-v1.md`

版本规则：

- 默认从 `v1` 开始
- 若根目录已存在同一“问题名 + 日期”的文件，则递增到 `v2`、`v3` ...
- 不覆盖已有总结文件

### Step 5：输出给用户

完成后对用户简要说明：

- 总结文件已生成
- 文件路径
- 本次总结覆盖的主问题
- 当前最关键的未完成项

## 文档结构模板

生成的 Markdown 文档默认使用下面结构：

```markdown
# [问题名]

> 日期：YYYY-MM-DD
> 版本：vN

## 本次在做什么
[用 1 到 2 段说明这次对话的目标、背景、核心问题]

## 当前结论
- [结论 1]
- [结论 2]

## 当前进度

### 已解决
- [已完成事项]

### 进行中
- [正在推进但尚未结束的事项]

### 未解决
- [仍未完成的事项]

## 未解决事项分析

### [事项 1]
- 现状：[当前卡点或缺口]
- 思路：[后续推进思路]
- 方案 A：[方案简述]
- 方案 B：[方案简述]
- 建议选择：[推荐方案及原因]

## 建议的下一步
1. [下一步 1]
2. [下一步 2]
3. [下一步 3]
```

可根据内容轻微调整，但必须保留：

- 在做什么
- 当前进度
- 已解决 / 未解决
- 未解决项思路
- 下一步建议

## 写作要求

- 优先准确，不要为了显得完整而编造进度
- 对“已解决”和“待验证”严格区分
- 简洁直白，适合交接给下一个继续处理的人
- 若引用代码结论，以实际文件内容和当前会话信息为准
- 若信息不足，明确写“待确认”而不是擅自下结论

## 适用触发词

以下表达出现时，优先使用本 Skill：

- 总结进度
- 会话总结
- 总结会话
- 生成交接文档
- 当前进度
- 现在做到哪了
- 帮我整理一下目前进展
- 帮我把这次聊天总结成文档
