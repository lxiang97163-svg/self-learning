# 商业点子探索与实施规格 Skill

## 本 Skill 解决什么

- 对用户**任意提到的商业点子**（产品、平台、服务、工具）做：**结构化探索 → 联网检索与归纳 → 风险与边界 → MVP 切片 → 可选技术栈建议 → 汇总成《实施规格》→ 组装一条返工最少的「开发用长 Prompt」**。
- 产出中须**明确标注**每一段结论属于哪一层：**Harness / Spec / Context / Agentic / Vibe**（见下文定义）。

## Agent 必须遵守

1. **检索优先**：在环境允许时，对「市场常见模式、竞品公开信息、合规关键词、技术选型对比」**使用联网搜索**（或等价工具），**禁止**仅凭训练记忆冒充「最新政策或竞品格局」。检索不到或不确定时，写「未验证 / 需人工核实」。
2. **非法律非投资建议**：合规与税务表述为**风险清单与待办**，用户须咨询律师/会计师；不对「稳赚、无风险」作承诺。
3. **诚实边界**：不存在「一条 Prompt 零 bug、永不再改」；最终交付的是**可开发、可验收、可回归**的规格与 Prompt，而非魔法。
4. **与用户工作区规则一致**：若项目另有 `agent-integrity` 等规则，不与之冲突；本 Skill 侧重商业与产品层。

---

## 五层定义（产出须标注）

| 层级 | 含义 | 典型产出 |
|------|------|----------|
| **Harness** | 红线、安全、法律边界、不做的事、数据与密钥纪律 | 禁运/牌照/未成年人/金融合规、免责声明结构、secret 不进对话 |
| **Spec** | 产品是什么、为谁、非目标、用户故事、验收、流程、数据模型、计费 | MVP 范围、页面/接口、状态机、成功指标 |
| **Context** | 约束条件与技术/组织环境 | 目标地区、预算、团队、必须对接的系统、栈偏好 |
| **Agentic** | 分阶段顺序与依赖 | 实现顺序、spike 顺序、测试与发布顺序 |
| **Vibe** | 品牌、文案气质、视觉偏好 | 语气、风格关键词（不替代 Spec） |

---

## 触发条件

用户自然语言包含但不限于：

- 「有个商业想法 / 创业点子 / 想做个 xxx」
- 「帮我探索可行性 / 检索整理 / 竞品 / 商业模式」
- 「输出 MVP 规格 / 实施文档 / 给开发的终极 Prompt」
- 「按 Harness Spec 分层整理」

若用户只给一句话点子，**先执行 Phase 0**（可反问 3～5 个极短问题），再进入检索；若用户拒绝回答，则用**占位符**继续并在《实施规格》中列出「待用户补充」。

---

## Phase 0：框题与约束（1 轮）

**目的**：把模糊想法收成可检索的问题陈述。

**Agent 动作**：

1. 用 3～6 个简短问题澄清（可选，用户可跳过）：目标用户？地区？是否涉及资金/跨境/医疗/金融/儿童？个人还是团队？大致时间预算？
2. 输出 **《问题陈述》**（200 字内）+ **《非目标》**（至少 3 条）+ **《成功长什么样》**（1 段）。

**层级标注**：主要为 **Spec**（边界）+ 少量 **Context**（环境）。

---

## Phase 1：发散与竞品/模式检索

**目的**：看见方案空间，避免一上来定栈。

**检索建议**（中英文混合，按点子改写关键词）：

- `[点子类型] marketplace business model` / `C2C platform MVP`
- `[地区] 监管 OR 牌照 OR 合规`（若业务敏感）
- `[核心动词] app competitors` / `alternatives to`
- `site: Product Hunt` 或 `site: crunchbase`（可选，作线索）

**Agent 输出**：

1. **竞品/类比品表格**（至少 3 个）：名称、模式、收费、目标用户、可借鉴点、差异点（**检索不到的列标注「未找到公开信息」**）。
2. **3 条可行路线**（例如：纯工具 / 撮合 / SaaS / 社区 / 硬件+软件），每条：**优点、缺点、适用前提、不适合谁**。
3. 每条结论旁用括号标注层级，如 `（Spec：范围）`、`（Harness：合规待核实）`。

**层级标注**：**Spec** 为主 + **Harness**（若涉及合规红线）+ **Context**（市场假设）。

---

## Phase 2：合规与风险 Harness（领域加粗）

**目的**：把「可能一票否决」的事提前列出。

**Agent 动作**：

1. 根据用户点子和地区，判断是否触发下列 **加粗域**；触发的域必须展开子检查表（检索 + 常识，**标注非法律意见**）：

| 加粗域 | 须额外检索的关键词示例 | Harness 产出 |
|--------|------------------------|--------------|
| 资金与用户资金 | `payment license`, `PSP`, `资金托管`, `支付牌照` | 是否涉及沉淀资金、是否需要牌照/备案 |
| 跨境与数据 | `GDPR`, `个人信息出境`, `cross-border data` | 数据存储地、出境、用户告知 |
| 医疗与健康 | `医疗器械`, `互联网医院`, `健康咨询 合规` | 是否构成诊疗建议、器械分类 |
| 金融证券 | `证券投资咨询 牌照` | 是否涉荐股、投顾资质 |
| 未成年人 | `未成年人保护`, `COPPA`, `儿童个人信息` | 年龄门、监护人同意 |
| 内容与 UGC | `平台责任`, `UGC 审核` | 审核义务、侵权通知 |
| 中国上架 | `小程序 类目`, `应用商店 资质` | ICP、软著、类目限制（仅作线索） |

2. 输出 **《Harness 风险与待核实清单》**：每条含：风险描述、严重度（高/中/低）、建议核实方式（律师/官方渠道/行业白皮书）、**本期 MVP 是否可降级规避**。

**层级标注**：**Harness** 为主。

---

## Phase 3：商业模式与单元经济（浅层）

**目的**：收费逻辑自洽，避免「有用户没钱」。

**Agent 输出**：

1. **可选收费设计** 至少 2 种：谁付费、何时付、与价值如何对应、优缺点。
2. **粗算**：若用户给了单价/抽成假设，做一页「假算表」（可简化）；未给则列**需向用户确认的数字**。
3. **防跳单/信任**（若适用）：平台内最小信任机制（评价、保证金 Phase 2、线下成交等）。

**层级标注**：**Spec**（规则）+ **Harness**（若涉资金沉淀）。

---

## Phase 4：MVP 切片与用户故事

**目的**：把「终极开发」收窄到可交付的一期。

**Agent 输出**：

1. **角色**（旅客/需求方/管理员等）与权限。
2. **用户故事列表**（每条含 **Given–When–Then** 一句验收）。
3. **本期必做 / 明确不做**（非目标与 Phase 2 列表）。
4. **核心实体草图**（实体名 + 关键字段，不必定死数据库）。

**层级标注**：**Spec** 为主。

---

## Phase 5：技术形态与栈（建议，非命令）

**目的**：给 Context，减少实现到一半换栈。

**Agent 动作**：

1. 在**不知用户偏好**时，给出 **2 个组合**对比：例如 Web+PWA vs 小程序 vs 原生（按业务选 relevant）；列：开发成本、上架/审核、支付、推送、国际化、SEO。
2. **推荐 1 个默认项**并说明假设（如「个人开发者、先验证需求则优先 Web」）。
3. 列出 **第一期必备技术组件**：认证、存储、消息、文件、搜索等（勾选式）。

**层级标注**：**Context** + 少量 **Spec**（交付物形态）。

---

## Phase 6：汇总《实施规格》Markdown

**目的**：单文档可交给下一轮「纯开发」会话。

**Agent 输出**一份完整 Markdown，**固定章节**如下（缺项写「待补充」）：

```markdown
# [项目代号] 实施规格 v0.1

## 1. 问题陈述与非目标
## 2. 用户与场景
## 3. 竞品与差异化（检索来源简述）
## 4. Harness：合规、安全、免责声明要点（非法律意见）
## 5. MVP 功能与验收（用户故事 + 验收标准）
## 6. 数据模型草案
## 7. 页面/模块/路由级清单（或 API 清单）
## 8. 关键流程（文字时序或编号）
## 9. 非功能需求（性能、语言、无障碍可写「本期不做」）
## 10. 开放问题与待用户决策
```

文末附 **层级索引**：列出各节主要属于 Harness / Spec / Context。

---

## Phase 7：组装「实施级终极 Prompt」（给开发 Agent）

**目的**：一条可复制、分层清晰、返工少的开发指令。

**Agent 必须**用下面**模板**填空（从 Phase 6 抽取），勿省略 Harness 与验收：

```text
【Harness｜缰绳】
- 法律与合规（非法律意见）：…
- 安全与隐私：密钥与环境变量；最小权限；…
- 本期范围锁死：不做 …；Phase 2：…

【Spec｜规格】
- 产品目标与非目标：…
- 角色与权限：…
- 核心实体与字段：…
- 页面/路由或 API：…
- 关键用户流程与验收：…

【Context｜上下文】
- 推荐技术栈与假设：…
- 部署/运行方式：…
- 语言与品牌：…

【Agentic｜实现顺序】
1）… 2）… 3）…（含测试与文档）

【验收】
- build/lint/test：…
- 手动验收步骤或 E2E 场景：…

【Vibe】
- UI/文案：…
```

并加一句：**「第一版目标为可验收，不承诺零缺陷；以本验收清单为完成定义。」**

---

## 会话衔接（多轮对话）

- 每轮结束输出 **《本轮结论摘要》** 5 行：已定 / 未决 / 下一轮建议。
- 若用户新开会话，首条消息可粘贴：**「继续商业点子探索，接上实施规格 v0.1，未决项：…」**

---

## 与用户其他 Skill 的关系

- 本 Skill **不替代** 具体领域 Skill（如股票复盘）；当用户同时提「先探索商业想法再写代码」时，**先完成本 Skill 至 Phase 6 或 7**，再切项目内技术 Skill。
- 若用户要求「把实施规格写入仓库」，默认路径建议：`docs/BUSINESS_SPEC_<slug>.md`（**仅当用户确认路径后写入**，避免擅自增加文件）。

## Agent 自检清单（执行结束前核对）

- [ ] 是否做过至少一轮**联网检索**（若环境允许）？关键结论是否区分「检索事实」与「推断」？
- [ ] **Harness** 是否覆盖点子对应的敏感域？
- [ ] **MVP 非目标** 是否写清，避免 scope creep？
- [ ] 是否输出 **Phase 6《实施规格》** 与 **Phase 7 终极 Prompt**？
- [ ] 是否声明非法律/非投资建议？
